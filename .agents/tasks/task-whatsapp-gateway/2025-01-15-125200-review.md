# WhatsApp Gateway: full-stack messaging platform with Baileys

A complete WhatsApp API gateway built as a TypeScript monorepo (pnpm workspaces): Express backend with SQLite, JWT auth, tiered subscriptions, WhatsApp session management via Baileys + baileys-antiban with pairing code auth, and a React/Vite/Tailwind frontend with landing page, auth flows, and dashboard. The architecture is a single-server deployment where the Express process holds live WebSocket connections to WhatsApp in memory.

**Watch for:** Hardcoded JWT secret in source fails open if `.env` is missing (confirmed). API keys are stored as plaintext in the database, making a DB dump a full credential leak (confirmed). The rate-limit middleware (`tierRateLimit`) is defined but never wired into any route (confirmed). Client expects camelCase fields but server returns snake_case — sessions and API keys will render broken data (confirmed).

## High-level view

The backend is a stateful singleton: `WhatsAppService` holds all active Baileys sockets in a `Map`, meaning horizontal scaling or process restarts during active connections will drop all sessions. Reconnection on startup (`restoreSessions`) re-establishes sockets for previously-connected sessions, but there's no mechanism to detect stale state if the process crashed without updating the DB.

Authentication uses JWT with bcrypt password hashing. The system defaults to a known secret string when no env var is present, which means a deployed instance without `.env` accepts tokens signed with the public default. There is no token rotation, no refresh token flow, and 7-day expiry means a leaked token has a wide window.

API keys are generated with good entropy (32 random bytes) but stored verbatim in SQLite. They function as bearer tokens for programmatic access, yet the key-based auth path isn't actually implemented — only JWT auth is checked in route middleware. The keys exist in the database but have no route that validates them for request authentication.

The tier system (free/pro/enterprise) enforces connection limits at connect time and message rate limits at send time. However, the generic `tierRateLimit` middleware that logs to `request_log` is never applied to any router, leaving the `request_log` table permanently empty. Rate limiting only happens inside the message-send handler via `checkMessageRateLimit`.

The frontend is a standard React SPA with token-based auth, axios interceptors for 401 handling, and a polling-based pairing code flow. The client assumes camelCase response fields (`phoneNumber`, `createdAt`) but the server returns raw SQLite column names (`phone_number`, `created_at`) — the sessions and API keys pages will display undefined values.

<summary>Issues (8)</summary>

1. **Hardcoded JWT secret default** — The fallback `'change-me-in-production'` in `auth.ts` and `auth.service.ts` means any deployment without `.env` is trivially exploitable. Remove the default or refuse to start without the env var.
2. **Plaintext API key storage** — API keys are stored unhashed in `api_keys.key`. Hash them (like passwords) and only display the full key once at creation time.
3. **Rate-limit middleware never mounted** — `tierRateLimit` in `middleware/rateLimit.ts` is not used by any router. Either wire it into the message/whatsapp routes or remove the dead code.
4. **API key auth path missing** — Keys are generated and stored but no middleware validates incoming requests by API key. The keys are useless for programmatic access.
5. **Client-server field name mismatch** — Frontend expects `phoneNumber`/`createdAt` (camelCase) but server returns `phone_number`/`created_at` (snake_case). Add a serialization layer or align the naming.
6. **Path traversal via phone number** — `getSessionDir` joins user-supplied `phoneNumber` into a filesystem path without sanitization beyond the 7-15 digit regex at the route level. If the service is called internally or the regex is bypassed, directory traversal is possible.
7. **No graceful shutdown** — The Express process doesn't handle `SIGTERM`/`SIGINT` to close Baileys sockets, flush WAL, or clear reconnect timers. Abrupt termination leaves sessions marked "connected" in DB with no live socket.
8. **Duplicate tier constants on client** — `Dashboard.tsx` hardcodes tier limits (`{ sessions: 1, messages: 100, keys: 1 }`) that must be kept in sync with the server's `seedTierLimits`. Fetch these from an API endpoint instead.

## JWT secret and auth posture

Both `middleware/auth.ts` and `services/auth.service.ts` declare:

```typescript
const JWT_SECRET = process.env.JWT_SECRET || 'change-me-in-production';
```

This constant is duplicated across two files, creating a secondary risk: if one is updated and the other isn't, token verification will silently fail or succeed depending on which path handles the request. More critically, the default value is committed to the repository. Any deployment that forgets to set the env var — or any environment where `dotenv` fails to load — will sign and verify tokens with a publicly-known secret.

The 7-day token expiry with no refresh mechanism means a compromised token grants week-long access. There's no logout/token-invalidation path; the client-side `logout` only removes localStorage.

## API key design gap

The `/api/keys` router generates keys with `crypto.randomBytes(32)` (good entropy) and stores the full key in SQLite. Two problems:

First, the keys are never validated anywhere. No middleware checks for an `X-API-Key` header or similar. The only auth gate is JWT-based (`authenticate` middleware). The keys can be created and revoked but serve no functional purpose.

Second, storing keys in plaintext means a database leak (SQLite file accessible, backup stolen, SQL injection in a future feature) exposes every key immediately. The standard practice is to store a hash and show the raw key only once at creation time.

## Session lifecycle and resource management

`WhatsAppService` is a singleton holding a `Map<string, ActiveConnection>`. Each connection holds a live Baileys WebSocket. This design means no horizontal scaling (you can't run two server instances without session-pinning), and process crashes leave `whatsapp_sessions.status = 'connected'` in the DB with no live socket.

The reconnect logic uses exponential backoff capped at 60s with a maximum of 5 attempts. After 5 failures, the connection is silently abandoned — the session stays in the `connections` Map with `status: 'disconnected'` forever (memory leak for long-running processes with many failed sessions). Only the `loggedOut` path triggers `connections.delete(key)`.

No graceful shutdown handler exists: `SIGTERM` doesn't trigger `cleanupConnection` for active sockets, doesn't flush the SQLite WAL, and doesn't clear pending reconnect timers.

## Path construction from user input

```typescript
private getSessionDir(userId: string, phoneNumber: string): string {
  return path.join(SESSION_PATH, userId, phoneNumber);
}
```

The route-level validator (`isValidPhoneNumber`) restricts to 7-15 digits, which prevents traversal at the HTTP boundary. But `initConnection` is also called from `restoreSessions` using `session.phone_number` from the database — if the DB is tampered with, or if a future code path bypasses the validator, the join is unsafe. A defense-in-depth approach would sanitize at the point of use.

## Rate limiting inconsistency

Two rate-limiting mechanisms exist:

1. `middleware/rateLimit.ts` — a generic per-request rate limiter that logs to `request_log`. **Never imported or used by any router.**
2. `whatsappService.checkMessageRateLimit` — checks `messages_log` count for today against `tier_limits.max_requests_per_day`. Used only in the `/api/messages/send` handler.

The `request_log` table will always be empty. The `/api/whatsapp/connect` endpoint has no rate limit — a user could spam connection attempts (each creating a Baileys socket and filesystem directory) without throttling beyond the tier's max-numbers check.

## Client-server contract mismatch

The frontend `SessionCard` component expects `session.phoneNumber` (camelCase), but the server returns raw SQLite column names (`phone_number`, snake_case). Same for `createdAt` vs `created_at`. The sessions page will show phone numbers as `undefined`.

The `ApiKeys` page expects `key.name` but the server never stores or returns a `name` field on API keys — `handleCreate` sends `{ name: newKeyName }` in the POST body but the server ignores it and returns `{ id, key, created_at }`. Every key will display as "Unnamed Key" regardless of what the user enters.

## Test coverage

Server-side: a single test file (`auth.service.test.ts`) covers registration, login, and user lookup.

Not tested: WhatsApp connection flow, reconnection logic, session restore, message sending, tier enforcement, rate limiting, API key creation/revocation, error responses from routes, validators, the client pairing code flow, auth context behavior.

Client-side: `App.test.tsx` renders static HTML strings in a MemoryRouter without mounting the actual App component.
