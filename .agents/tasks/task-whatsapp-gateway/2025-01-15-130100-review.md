# WhatsApp Gateway: v2 review after security and correctness fixes (v2)

The eight issues from v1 have been addressed: JWT secret now throws on startup without `JWT_SECRET` in production, API keys are SHA-256 hashed with the raw key shown only at creation, rate-limit middleware is mounted on both whatsapp and messages routes, API key auth middleware validates `X-API-Key` headers for programmatic access, responses are serialized to camelCase, phone numbers are sanitized to digits-only in path construction, graceful shutdown handlers close sockets and the DB, and the Dashboard fetches tier limits from `GET /api/tiers`.

**Watch for:** Creating a second API key will crash with a UNIQUE constraint violation on the `api_keys.key` column because all keys now insert `''` as the raw key value (confirmed). The Pricing page advertises "Unlimited sessions" for Enterprise but the database enforces a cap of 20 (confirmed). The `request_log` table grows unbounded with no pruning — every authenticated request to `/api/whatsapp` or `/api/messages` inserts a row that is never cleaned up (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The API key hashing fix introduced a data-integrity bug: the schema declares `key TEXT UNIQUE NOT NULL` on the `api_keys` table, but the insert now writes `''` for every key. SQLite's UNIQUE constraint treats two empty strings as duplicates, so any user who creates more than one API key will get a hard failure. The migration adds `key_hash` and `name` columns but doesn't drop or relax the `key` column's UNIQUE constraint.

Rate limiting now fires on all authenticated `/api/whatsapp` and `/api/messages` requests, counting them against `max_requests_per_day`. Because `GET /sessions` passes through the same limiter, dashboard navigation and session listing consume from the same budget as message sending. A free-tier user gets 100 requests/day total — which means loading the dashboard (3 API calls) burns 3% of their daily budget.

The Pricing page still hardcodes tier feature lists (sessions, messages, API keys per tier) rather than fetching them from `/api/tiers`. The Dashboard properly fetches limits from the server, but Pricing.tsx claims "Unlimited sessions" and "Unlimited API keys" for Enterprise while the database enforces 20 sessions and has no API key count enforcement at all.

1. **UNIQUE constraint on `api_keys.key` breaks multi-key creation** — The schema declares `key TEXT UNIQUE NOT NULL` but the insert writes `''` for every key. Remove the UNIQUE constraint or store a unique placeholder (e.g., the key ID) in that column. (confirmed)

4. **Read-only API calls consume rate-limit budget** — `GET /api/whatsapp/sessions` and `GET /api/messages/history` pass through `tierRateLimit`, decrementing the same daily counter as mutations. Separate read and write budgets, or exempt GET requests from the rate limiter. (confirmed)

## UNIQUE constraint collision on API key creation

The `api_keys` table schema (in `db/schema.ts`) declares:

```sql
key TEXT UNIQUE NOT NULL
```

After the hashing fix, `routes/api-keys.ts` inserts:

```typescript
db.prepare(
  'INSERT INTO api_keys (id, user_id, key, key_hash, name) VALUES (?, ?, ?, ?, ?)'
).run(id, userId, '', keyHash, name || '');
```

The first key insert succeeds (empty string is valid for NOT NULL). The second insert fails because `''` already exists and violates UNIQUE. The migration in `db/migrations.ts` adds `key_hash` and `name` columns but never alters the `key` column constraint. The fix is either to drop the UNIQUE constraint on `key` (since `key_hash` now serves as the lookup column), or to store a unique placeholder value (like the key's UUID) in the `key` field.

## Rate-limit scope overshoot

`tierRateLimit` is applied at the router level for both `/api/whatsapp` and `/api/messages`:

```
whatsapp.ts:  router.use(authenticate); router.use(tierRateLimit);
messages.ts:  router.use(authenticateOptionalJwt); router.use(authenticateApiKey); router.use(tierRateLimit);
```

Every request — including `GET /sessions`, `GET /sessions/:id/status`, and `GET /messages/history` — inserts into `request_log` and counts toward `max_requests_per_day`. The Dashboard alone makes 3 parallel requests on mount (`/whatsapp/sessions`, `/keys`, `/tiers`) and the first two pass through rate-limited routers. A free-tier user with 100 requests/day will exhaust their budget after roughly 33 dashboard page loads without sending a single message.

The `request_log` table also has no index on `(user_id, timestamp)`. The daily-count query will table-scan as the log grows. And since rows are never deleted, the table accumulates indefinitely.

## Pricing page still hardcodes limits

The Dashboard correctly fetches from `GET /api/tiers` and displays dynamic limits. The Pricing page does not — it declares a static `tiers` array with feature strings like "1 WhatsApp session", "Unlimited sessions", etc. Enterprise says "Unlimited sessions" but `seedTierLimits` inserts `max_numbers = 20`. If the database limits change, the Pricing page will show stale numbers. The v1 issue was specifically about the Dashboard, which is now fixed, but the Pricing page has the same problem in a more visible way.

## API key auth asymmetry

The `/api/messages` routes use `authenticateOptionalJwt` + `authenticateApiKey`, so API key holders can send messages and fetch history. But `/api/whatsapp` uses strict `authenticate` (JWT only), so the same API key cannot list sessions or initiate connections. A programmatic user who authenticates solely via API key can send messages through a session but cannot manage sessions. This might be intentional (session management requires interactive login), but it's worth documenting explicitly since the API Keys page tells users to "use the API" without mentioning this limitation.

Additionally, when a user provides an expired Bearer token to the messages route, `authenticateOptionalJwt` silently swallows the error and falls through to API key auth. If no `X-API-Key` header is present, the 401 response says "Provide a Bearer token or X-API-Key header" — obscuring the real problem (expired token). This is a UX gap rather than a security issue.
