# WhatsApp Gateway: final verification of v2 fixes (v3)

All five issues from v2 have been resolved. The UNIQUE constraint on `api_keys.key` is gone (the column now uses `NOT NULL DEFAULT ''` without uniqueness). The `request_log` table has a composite index on `(user_id, timestamp)` and hourly cleanup via `cleanupRequestLog`. The Pricing page fetches tier limits from `GET /api/tiers` and builds feature lists dynamically. Rate limiting (`tierRateLimit`) is applied only to mutation endpoints (`POST /connect`, `DELETE /sessions/:id`, `POST /send`) â€” read endpoints pass through unmetered. The `getJwtSecret` function lives in `utils/jwt.ts` and is imported by both `middleware/auth.ts` and `services/auth.service.ts`.

**Watch for:** The Landing page still advertises "Unlimited sessions" for Enterprise while the database seeds `max_numbers = 20` â€” this is the public marketing page, not the authenticated Pricing page that was fixed (likely, low severity). No index exists on `api_keys.key_hash`, which is the lookup column for every API-key-authenticated request (confirmed, low severity at expected scale).

**Verdict**: APPROVED

## High-level view

Rate limiting is now surgically applied per-route on mutations only. GET endpoints are exempt, so dashboard navigation no longer burns daily budget.

The Pricing page fetches dynamically from `/api/tiers`. The Landing page retains static copy ("Unlimited sessions" for Enterprise) since it's a marketing surface for unauthenticated visitors â€” a minor copy/DB mismatch but not a functional bug.

`POST /api/messages/send` passes through both `tierRateLimit` (counting in `request_log`) and `checkMessageRateLimit` (counting in `messages_log`). Both use the same `max_requests_per_day` threshold but against different tables. The effective daily message limit is `max_requests_per_day minus connection/disconnection operations` â€” acceptable for a v1 rate-limiter but worth knowing.

## Double rate-limit check on message send

`POST /api/messages/send` passes through both `tierRateLimit` (middleware, counting against `request_log`) and `whatsappService.checkMessageRateLimit` (checking `messages_log`). Both use `max_requests_per_day` from `tier_limits` as their threshold but count against different tables. The middleware counter also counts `/connect` and `/disconnect` operations, so a free-tier user with 100/day who connects a session has 99 remaining for messages. The redundancy means either counter can reject the request first â€” the middleware's `request_log` count will always be >= the `messages_log` count, making `checkMessageRateLimit` unreachable as a rejection path unless the user only sends messages and never connects/disconnects.

## Landing page copy drift

`Landing.tsx` defines a static `tiers` array with hardcoded strings. Enterprise claims "Unlimited sessions" and "Unlimited API keys". The database seeds 20 max sessions for enterprise; API key counts aren't enforced at all. The discrepancy exists only on the unauthenticated marketing surface â€” the authenticated Pricing page dynamically shows "20 WhatsApp sessions".
